# SCQR Editorial System — AGENTS.md

> Этот документ — редакционный контракт. Его читают и исполняют все AI-агенты, работающие с SCQR: Claude Code (через `CLAUDE.md`, который ссылается на этот файл), будущий Codex (через собственный AGENTS.md-механизм) и любые другие гарнизы. Для специфики каждого harness — см. соответствующий отдельный документ.

## Миссия

SCQR — русскоязычное аналитическое издание об искусственном интеллекте. Мы помогаем руководителям, аналитикам, исследователям и ИТ-специалистам быстрее понимать, как меняется рынок ИИ и какие решения становятся возможны/необходимы. Мы не гонимся за скоростью и объёмом — мы делаем меньше материалов, но сильнее.

## Ключевой принцип

Главный объект редакционной системы — **не статья, а рецензия владельца на пакет сигналов**.

1. Утром и вечером система собирает сигналы с источников, группирует их в md-пакет.
2. Владелец читает пакет и пишет свободную рецензию: где его зацепило, что важно, какой угол, что отложить.
3. Интерпретатор разбирает рецензию на JSON-`review_actions` по каждому сигналу (`write | defer | archive | keep_context | combine | series_candidate`) с полями `thesis`, `must_include`, `must_avoid`.
4. Для действий `write` создаются `article_jobs`.
5. Writer → editor → factchecker → visual-briefer → publish. Владелец апрувит финальное.

Никакая статья не начинается без явного действия `write` в `review_actions`. Это защита от дрейфа.

## Жёсткие правила

1. **Язык публикаций — только русский.**
2. **Главный вывод статьи приходит от владельца через рецензию**, а не от агента. Writer формулирует детали, но тезис — владельца.
3. **Слабые материалы не публикуются.** Лучше `defer` на неделю, чем выпустить пустое.
4. **Ни один агент не пушит напрямую в `main`.** Только через ветку `article/<slug>` + ручной PR владельца.
5. **Форумы и агрегаторы без редактуры — не источник.** Только первоисточники, регуляторы, крупные лабы, качественные медиа и профильные аналитики.
6. **Фактчек обязателен.** Любая цифра, дата, имя, цитата — с подтверждением из первичного источника. Без подтверждения — вычёркивается. Никаких «эксперты говорят» или непонятных оборотов.
7. **Визуал и инфографика требуют отдельного апрува.**
8. **Все сигналы сохраняются в памяти** (таблица `signals`), даже если по ним не пишем. `decision_log` фиксирует причину.
9. **Стилистика — жёстко по `docs/style-guide.md`.** Запрещены: bullet-points в теле статей (разрешены в прямых списках сравнений); обороты «мы считаем», «по нашему мнению»; англицизмы там, где есть русский эквивалент («кейс» → «пример», «челлендж» → «вызов»); выдуманные цитаты и цифры.
10. **Не трогаем `site/src/content/posts/*.md` без явной `article_job`** из `pnpm scqr produce`. 65 существующих статей — ground truth; правки — только по явному запросу владельца.

## Архитектура: control plane и data plane

- **Data plane** — TypeScript CLI `pnpm scqr <command>` в `engine/`. Детерминированные атомарные операции: БД, сборка md-пакета, валидация, сохранение статьи, git-операции. Без LLM-вызовов.
- **Control plane** — активная сессия AI-агента (сейчас Claude Code, в будущем Codex). Принимает решения, вызывает subagents через механизм harness (Task tool для Claude Code, `codex exec` для Codex), читает/пишет файлы в `.scqr/jobs/<id>/`, вызывает CLI для фиксации в БД.
- **Контракт между ними** — файлы в `.scqr/jobs/<id>/` (input.json, draft.md, edited.md, factcheck.json, visual-brief.md) и `.scqr/packs/*.md`. Агенты не передают данные через stdin/stdout — только через файлы, чтобы любой harness мог подключиться.

Схема:

```
sources → signals → clusters → packs (md)
                                 ↓
                         владелец читает, пишет рецензию (.md)
                                 ↓
                         review-interpreter (agent) → review_actions (JSON)
                                 ↓
                         article_jobs (БД) + .scqr/jobs/<id>/input.json
                                 ↓
              writer → editor → factchecker → visual-briefer (agents)
                                 ↓
                         site/src/content/posts/<slug>.md + visual-brief.md
                                 ↓
                         publish (CLI: git branch + commit) → владелец открывает PR
                                 ↓
                         Vercel deploy → decision_log + memory
```

## Агенты (8)

| Name | Роль | Тип |
|---|---|---|
| `signal-collector` | Обходит реестр источников, нормализует, дедуплицирует, кластеризует, пишет в `signals`. | Рутина |
| `review-interpreter` | Разбирает свободный текст рецензии владельца на JSON `review_actions`. | Критичный |
| `writer` | Пишет статью по `input.json` (тезис, must_include, evidence, style-corpus). | Критичный |
| `editor` | Приводит к `docs/style-guide.md` и `docs/editorial-rules.md`. | Рутина |
| `factchecker` | Проверяет все цифры/даты/имена/цитаты по первоисточникам. | Критичный |
| `visual-briefer` | Формирует `visual-brief.md` (cover + инфографика). Не генерирует изображения. | Критичный |
| `source-scout` | Deep-research по новым источникам (еженедельно). | Рутина |
| `retro-reviewer` | Еженедельная ретроспектива `decision_log`. | Рутина |

Полные промпты — в `.claude/agents/*.md` (для Claude Code) или `.codex/agents/*.toml` (для Codex, добавим позже).

## Skills (12)

Каждый skill — повторяемая процедура, которую агент выполняет целиком. Расположены в `.claude/skills/<name>/SKILL.md`:

1. `collect-signals` — запуск коллекторов, нормализация, запись в БД.
2. `build-signal-pack` — формирование md-пакета из новых сигналов.
3. `interpret-review` — разбор рецензии на JSON actions.
4. `assemble-evidence` — сбор доказательной базы для writer из signals + source pages.
5. `write-article` — производство draft по style-corpus и input.json.
6. `edit-article` — редактура под стайл-гайд.
7. `factcheck-article` — проверка фактов, возврат JSON-отчёта.
8. `brief-cover` — визуальный бриф для Codex-operator.
9. `brief-infographic` — SVG-инфографика (инструкция или готовый SVG).
10. `publish-article` — валидация + git-ветка + коммит.
11. `review-sources` — audit реестра источников (доступность, качество).
12. `weekly-retro` — выгрузка decision_log и оформление memory-записи.

## Memory

После каждой публикации и еженедельно (retro-reviewer) дописываем запись в таблицу `memory` и в `docs/memory/YYYY-WW.md`:

- что сработало (какой угол зашёл, какой source дал сигнал),
- что не сработало (какой сигнал выбросили и почему, какие темы переоценили),
- повторяющиеся паттерны (какие сюжетные линии складываются, какие темы возвращаются),
- решения по источникам (кого добавить, кого заморозить).

Memory — это не архив, это подсказка следующему пакету. Retro-reviewer и score.ts читают её при ранжировании новых сигналов.

## Harness-специфика

- **Claude Code** — `CLAUDE.md` в корне + `.claude/agents/` + `.claude/skills/`. Модели: `opus` для критичных, `sonnet` для рутины. Подробнее — `docs/harness-claude-code.md`.
- **Codex** — будет добавлено: `.codex/agents/*.toml` + automations. Модели: `gpt-5.5` для критичных, `gpt-5.4` для рутины. Подробнее — `docs/harness-codex.md` (план).
- **Различия**: только в том, **как** агент спавнится и с какой моделью. Контракт «что должно получиться на выходе» и файловая разметка `.scqr/jobs/<id>/` — едины. Это намеренно: перенос между harness сводится к генерации новых spec-файлов из одного манифеста.

## Запрещено для всех агентов

- Прямой push в `main` (всегда через feature-ветку).
- `git commit --no-verify`, `git push --force`, `git reset --hard` на любых тречных ветках.
- Мутация `site/src/content/posts/*.md` без активной `article_job` в БД.
- Правка `site/src/assets/editorial/` без активной `visual_brief` в `.scqr/visual-queue/`.
- Запуск `npm install` / `pnpm add` без явного одобрения владельца.
- Публикация без прохождения `factchecker` (factcheck.json должен содержать `status: "pass"` по всем пунктам).
- «Молчаливое» игнорирование сигнала — любое отклонение должно быть записано в `decision_log`.
