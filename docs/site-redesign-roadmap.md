# SCQR Site Redesign Roadmap

## Цель
Довести SCQR до состояния цельного редакционного продукта: сильная первая полоса, читаемая статья, понятные рубрики, живые сюжетные страницы, аккуратная графика и стабильная сборка для Vercel.

## Дизайн-система
- Основа интерфейса: светлая бумажная поверхность, плотная редакционная сетка, кислотно-зелёный акцент только в местах смысла.
- Главная должна выглядеть как первая полоса издания, а не как лендинг или каталог карточек.
- Каждая страница обязана иметь ясный ведущий блок, вторую навигационную зону и спокойный нижний хвост.
- Визуальный язык статьи и архива должен быть единым: одни и те же метки, один ритм типографики, одна логика карточек.

## Структура страниц

### Главная
- 1 ведущий материал.
- 4 визуальные карточки первой линии.
- Блок сюжетов.
- Две ленты: новости и аналитика.
- Полоса колонок.
- Быстрая навигация по рубрикам.

### Статья
- Верх статьи обязан сразу показывать рубрику, формат, дату, время чтения и заголовок.
- Справа на desktop: вывод SCQR, основание публикации, темы и сюжетная навигация.
- Основной текст держится в ширине 680–760px.
- Внутренние графики и большие изображения могут выходить в более широкий режим, но не ломать ритм текста.
- Внизу обязательны связанные материалы и переход по сюжету или рубрике.

### Архив
- Архив делится по форматам: новости, аналитика, колонки.
- Каждая секция должна быть пригодна для быстрого просмотра без фильтров на JavaScript.
- Карточка архива обязана показывать рубрику, формат и время чтения.

### Рубрика
- Верх рубрики: название, описание, объём материалов, RSS.
- Далее: ведущий материал, визуальный блок, затем архивный поток.
- Внизу — переход в соседние рубрики.

### Сюжет
- Сюжет должен читаться как развивающаяся история во времени.
- Наверху — краткое объяснение линии, первый материал, текущая точка и общий вывод.
- Ниже — таймлайн с датами и карточками.

### Тема
- Тема остаётся лёгкой подборкой, а не второй рубрикой.
- Минимум декоративной сложности.
- Акцент на быстром переходе к материалам.

## Изображения и графика
- Канон визуального стиля хранится в `docs/editorial-image-system.md`.
- Визуальные блоки главной используют только материалы с качественными обложками.
- В архиве допускается двухуровневая система: `premium`-обложки для витрины и `secondary`-обложки для нормализованного legacy-слоя.
- `status: "ready"` для новых материалов предполагает заполненные `heroImage`, `sourceNote`, `topics`, `readingTime`.
- Внутренние графики остаются SVG, но должны использовать единые поля, подписи и сетку.

## Контентные правила
- `Новации` — только новые запуски, стартапы, анонсы и релизы.
- Политические новости вне прямого влияния на рынок ИИ не должны попадать в публичную витрину.
- Для спорных или нерелевантных новостей использовать внутренний флаг `archive-noise`, чтобы хранить их в архиве как источник, но не пускать в публичную поверхность сайта.
- У каждого ready-материала должен быть ясный авторский вывод уже в первых абзацах.
- Англицизмы допустимы только когда они действительно добавляют точность.

## Технические критерии
- `npm run validate`
- `npm run check`
- `npm run build`
- Корректные `canonical`, `og:image`, `twitter:image` и `article:published_time` на страницах статей.
- Рабочий sitemap и общий RSS, плюс RSS рубрик.

## Ближайшая очередь
1. Заменить заглушки на AI-обложки сначала для первой волны, затем для исторического бэкфилла.
2. Подтянуть ещё 2–3 сюжетные поверхности на главную по данным, а не вручную.
3. Довести OG-изображения до отдельной редакционной системы для соцсетей.
4. Прогнать ручную мобильную ревизию после каждого большого визуального изменения.
